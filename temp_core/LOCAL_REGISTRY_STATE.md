# Local Registry State Model

The Ledger local registry is a governance state model, not a browser storage detail.

The Registry should become the local governance operations center for authority lifecycle posture, replay readiness, diagnostic awareness, supersession chains, and publication evidence. This document freezes the Ledger-side state model before more Registry code is added.

## Core Rule

Do not make `localStorage` the model.

The model is:

```text
Registry State Model
  -> Storage Adapter
  -> localStorage now
  -> SQLite later
```

`localStorage` is only the first adapter. Future implementations may use SQLite, a file-backed store, or another local durable store without changing the canonical object semantics.

Frontend views should render registry state. They should not derive governance meaning from ad hoc browser storage shapes.

## Object Relationship

```text
authority_workspace_projection.v1
    current working view of one authority draft/release state

authority_registry_entry.v1
    durable local record of an exported/registered authority

authority_lifecycle_event.v1
    append-only lifecycle event attached to an authority lineage

publication_receipt.v1
    evidence that a bundle export occurred with hashes and review posture

diagnostic_rollup.v1
    deterministic summary of advisory governance findings for a specific authority state
```

The practical relationship is:

```text
authority_workspace_projection.v1
  may become
authority_registry_entry.v1
  contains many
authority_lifecycle_event.v1
  references optional
publication_receipt.v1
  includes
diagnostic_rollup.v1
```

## State Model Boundary

The Registry State Model owns local Ledger state about:

- current authority workspace posture
- registered authority lifecycle records
- publication receipt availability
- replay readiness posture
- continuity posture
- supersession and revocation history
- advisory diagnostics rollups
- local governance event timelines

It must not own:

- Guard admissibility outcomes
- Cloud ingestion state
- Cloud review queues
- runtime execution evidence mutation
- account, organization, billing, or RBAC state
- inferred governance meaning outside Ledger semantic artifacts

## authority_workspace_projection.v1

`authority_workspace_projection.v1` is the current working view of one authority draft or release state.

It is generated by Ledger semantic and UI projection code from canonical artifacts such as `authority_contract.v1`, `governance_impact_preview.v1`, `authority_bundle.v1`, diagnostics, registry state, and optional publication receipt state.

It is not a durable registry record by itself.

Purpose:

- provide one canonical UI projection for Overview, Publication, Diagnostics, and Registry surfaces
- prevent page-specific state drift
- expose operator-facing governance posture without requiring every page to reconstruct meaning

Suggested shape:

```json
{
  "schema_version": "authority_workspace_projection.v1",
  "authority_ref": "treasury-policy@2.1.0",
  "lifecycle_posture": "draft",
  "review_state": "impact_reviewed",
  "export_state": "not_exported",
  "registration_state": "not_registered",
  "operational_change": "Executions with amount > $250,000 require treasury-governance review.",
  "continuity_posture": "Resumed workflows require continuity revalidation when governance posture changes.",
  "lifecycle_effect": "Revoked authorities invalidate resumed execution continuity.",
  "publication_meaning": "treasury-policy@2.1.0 governs transfer funds for Corporate Treasury Transfer System.",
  "registry_posture": "Bundle ready to export. Authority not registered locally.",
  "replay_posture": "Export will create publication receipt evidence that future replay can bind against.",
  "diagnostic_rollup": {},
  "timeline": []
}
```

Notes:

- `authority_workspace_projection.v1` is read-model state.
- It should be cheap to regenerate.
- It should not be treated as publication evidence.
- It should be stable for identical inputs except for explicitly timestamped workflow metadata.

## authority_registry_entry.v1

`authority_registry_entry.v1` is the durable local record of an exported or registered authority.

It represents the local lifecycle state for an authority reference and binds semantic artifacts, publication evidence, and lineage metadata.

Purpose:

- make registered authorities visible in the local Registry
- preserve lifecycle state across UI sessions
- bind the authority to immutable semantic inputs and receipt evidence
- support supersession, revocation, lineage, and replay readiness surfaces

Suggested shape:

```json
{
  "schema_version": "authority_registry_entry.v1",
  "registry_id": "registry-...",
  "authority_ref": "treasury-policy@2.1.0",
  "status": "registered",
  "published_at": "2026-05-26T18:00:00Z",
  "superseded_by": null,
  "governed_resource": "Corporate Treasury Transfer System",
  "governed_action": "transfer funds",
  "continuity_posture": "resume revalidation and revocation invalidation",
  "escalation_threshold": "amount > 250,000",
  "semantic_integrity_posture": "compatible",
  "contract_hash": "sha256:...",
  "immutable_inputs": {},
  "lineage": {},
  "publication_receipt": {},
  "diagnostic_rollup": {},
  "lifecycle_timeline": [],
  "artifacts": {}
}
```

Status values should initially remain narrow:

- `registered`
- `superseded`
- `revoked`

Additional values should be additive and should not change the meaning of existing status values.

## authority_lifecycle_event.v1

`authority_lifecycle_event.v1` is an append-only lifecycle event attached to an authority lineage.

It records local governance lifecycle movement. It is not an execution event and must not represent Guard admissibility.

Purpose:

- show authority lifecycle history
- support Registry timelines
- support supersession chains
- preserve local publication posture
- provide audit-friendly local event history

Suggested shape:

```json
{
  "schema_version": "authority_lifecycle_event.v1",
  "event_id": "event-...",
  "authority_ref": "treasury-policy@2.1.0",
  "event_type": "registered",
  "timestamp": "2026-05-26T18:00:00Z",
  "hash": "sha256:...",
  "detail": "Authority registered locally with authority_bundle.v1 and publication_receipt.v1.",
  "actor": "governance-ledger-ui",
  "causality": {
    "previous_event_id": "event-...",
    "publication_receipt_hash": "sha256:..."
  }
}
```

Initial event types:

- `drafted`
- `impact_reviewed`
- `bundle_exported`
- `registered`
- `superseded`
- `revoked`

Lifecycle events are append-only. Corrective events should be added as new events rather than mutating prior events.

## publication_receipt.v1

`publication_receipt.v1` is evidence that a bundle export occurred with hashes and review posture.

It is the local publication evidence object. It does not prove Cloud ingestion and does not change execution outcomes.

Purpose:

- bind exported authority bundle to immutable hashes
- record publication timestamp
- record readiness confirmations
- record publication notes
- support replay-ready local evidence posture

Required meaning:

```json
{
  "schema_version": "publication_receipt.v1",
  "authority_ref": "treasury-policy@2.1.0",
  "published_at": "2026-05-26T18:00:00Z",
  "bundle_hash": "sha256:...",
  "manifest_hash": "sha256:...",
  "semantic_artifact_hashes": {},
  "readiness_confirmations": {},
  "publication_notes": [],
  "lineage_continuity": {},
  "compatibility_posture": {},
  "receipt_hash": "sha256:..."
}
```

Non-goals:

- does not publish to Cloud
- does not approve authority
- does not execute authority
- does not call Guard
- does not mutate replay or evidence

## diagnostic_rollup.v1

`diagnostic_rollup.v1` is a deterministic summary of advisory governance findings for a specific authority state.

It should summarize diagnostics without becoming policy scoring, rejection logic, or admissibility evaluation.

Purpose:

- display Registry-level diagnostic posture
- support Overview rollups
- support stale review and governance quality warnings
- preserve diagnostic state for a registered authority

Suggested shape:

```json
{
  "schema_version": "diagnostic_rollup.v1",
  "authority_ref": "treasury-policy@2.1.0",
  "source_hash": "sha256:...",
  "generated_at": "2026-05-26T18:00:00Z",
  "finding_count": 2,
  "warning_count": 1,
  "info_count": 1,
  "domains": ["approval", "lifecycle"],
  "highest_severity": "warning",
  "findings": [
    {
      "code": "GQ005",
      "title": "Lifecycle Ambiguity",
      "severity": "warning",
      "domain": "lifecycle"
    }
  ]
}
```

Diagnostics remain advisory. They must not block publication or become Guard.

## Storage Adapter Boundary

All local persistence should pass through a storage adapter boundary.

Conceptual interface:

```text
load_workspace_projection(authority_ref)
save_workspace_projection(projection)

list_registry_entries()
get_registry_entry(authority_ref)
save_registry_entry(entry)

append_lifecycle_event(authority_ref, event)
list_lifecycle_events(authority_ref)

save_publication_receipt(authority_ref, receipt)
get_publication_receipt(authority_ref)

save_diagnostic_rollup(authority_ref, rollup)
get_diagnostic_rollup(authority_ref)
```

The first implementation may use browser `localStorage`, but the application code should treat that as an adapter implementation detail.

The future SQLite adapter should be able to preserve the same versioned objects without changing page rendering logic or semantic derivation logic.

## View Rules

Overview, Registry, Diagnostics, and Publication should render views over canonical governance state.

They should not become independent state machines.

Rules:

- Overview renders workspace projection plus registry rollups.
- Registry renders durable registry entries and lifecycle events.
- Diagnostics renders diagnostic rollups and detailed diagnostic artifacts.
- Publication renders workspace projection, receipt posture, and export/registration actions.
- No page should independently infer governance meaning from raw form fields.
- No page should treat storage adapter keys as canonical object structure.

## Determinism

The Registry State Model must preserve Ledger's deterministic guarantees:

- semantic meaning is generated by Ledger semantics
- diagnostic rollups are deterministic for identical authority state
- lifecycle events are append-only
- hashes bind immutable publication inputs
- storage adapters do not change object meaning
- UI pages render state; they do not create governance meaning

## Evolution Rules

All local registry objects are versioned.

Changes should be:

- additive where possible
- schema-versioned when meaning changes
- deterministic for identical inputs
- adapter-independent
- explicit about timestamped and user-authored fields

Do not overload `authority_registry_entry.v1` with temporary UI state. Temporary state belongs in the workspace projection or in an adapter-specific session object.
